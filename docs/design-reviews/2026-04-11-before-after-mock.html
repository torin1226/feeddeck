1. Settings Page Crash Recovery P0 5th consecutive run

Settings has crashed on every audit since Run 5. The error boundary catches it, but the recovery UX is generic and unhelpful. The user sees a code error message and has no idea what went wrong or what to do about it.

5. Global: API Timeout & Retry Infrastructure P1 New pattern

Zero API calls have timeouts. Zero have automatic retry (except queue polling). A single slow endpoint can freeze any page indefinitely. This is the root infrastructure gap behind most resilience issues.

Current: Raw fetch, no timeout
// homeStore.js - fetchHomepage()
const res = await fetch(`${API}/homepage`)
// No timeout
// No retry
// No AbortController
// Waits forever
// Same pattern in feedStore, libraryStore,
// settingsPage (6+ call sites)
Proposed: fetchWithResilience() utility
// utils/resilientFetch.js
const res = await fetchRetry(`${API}/homepage`, {
  timeout: 8000,     // 8s max
  retries: 2,       // 2 retries
  backoff: 'exponential',
  onRetry: ({ attempt }) =>
    set({ loadHint: `Retry ${attempt}/2...` }),
})
What to notice
1. Single utility replaces 6+ raw fetch() calls across the app
2. 8s timeout prevents indefinite hangs
3. Exponential backoff (1s, 2s, 4s) for server recovery time
4. onRetry callback lets UI show "Retry 1/2..." in real time
5. ~30 lines of code for the utility. 15 min to wire into all stores.
